## 核心观点

这篇文章是 Tw93 给非技术背景的产品经理、业务和运营人员分享的 AI Coding 上手指南。核心观点：**AI Coding 门槛没想象的高，会用豆包对话框 AI 的人花点时间也能上手**，剩下的就是慢慢习惯把执行权交给 AI。

### 关键要点

1. **命令行不是程序员的专利** - Kaku 终端为 AI Coding 设计，内置 AI 辅助和修复建议
2. **补点技术知识很重要** - 不需要会写代码，但要知道框架、终端命令、Git、VS Code、Chrome 开发者工具的基本概念
3. **订阅注意事项** - 美国 IP 环境、Gmail 老账号、实体卡付款、不要直接包 Max
4. **Claude Code 最适合这类任务** - 做原型和内部小工具、处理 CSV/报表、文档整理、信息提炼
5. **CLAUDE.md 是核心** - 项目背景文件，每次启动都会认真读，相当于给 AI 写的交接文档
6. **需求描述越精确，AI 越少分叉跑偏** - 提供具体约束条件、数字化的验收标准

## 第一道坎是命令行

不写代码的同学习惯了对话框 AI，第一次装 Claude Code 会有点不适应。以前是来回搬运的过程——你描述需求、AI 生成代码、你复制粘贴到别处去试。现在变成 Claude Code 直接在终端运行，搬运这一步省掉了。

### 推荐工具：Kaku

- 深色浅色跟着系统走

- 内置 AI 辅助：命令跑报错了会自动给修复建议
- 记不住命令在前面加 `#` 写中文也能生成

## Claude Code 适合什么样的活

Claude Code 最不一样的地方：**模型能力本身很不错 + 代码实现把 Harness 玩到极致**。它先扫一遍 CLAUDE.md 和目录结构摸清楚上下文，然后跨文件改代码、跑命令、看报错、再改，自己全部完成。

### 实际适合的任务类型

1. **做原型和内部小工具** - 把需求和展示逻辑说清楚，第二天就能跑起来一版
2. **处理 CSV、做销售报表** - 分组和计算逻辑写明白几分钟出结果
3. **文档活** - 几十页合同提炼条款、对比版本差异
4. **信息提炼** - 给一堆链接或 PDF 让它从特定视角提炼信息

### 关于 Cowork

Anthropic 官方出的桌面应用，能直接读写下载和文档目录。把收据截图拼成报销表这种活，说一句话它也能给你干好。

### 一个重要原则

**写代码这件事上，模型快不快不重要，准不准才重要。** 它 10 分钟跑完然后你花 20 分钟 debug，远不如它 20 分钟跑完直接能验收来的舒服。

## 做一个只给你一个人用的软件

《纽约时报》专栏作家 Kevin Roose 提过一个概念叫 **software for one**：你不需要做给一百万人用的 App，可以做只给你一个人用的软件。

- **Stash** - 整理链接
- **LunchBox Buddy** - 给孩子准备便当

对你来说，可能是把语音批注转成会议纪要的工具，或者是每天提醒你三件事的小仪表盘。这种东西反而是产品和业务的同学最容易做成，毕竟只有你最懂自己每天的麻烦在哪。

## 用 OpenCLI 把网页操作变成命令

[OpenCLI](https://opencli.info/) 是 Tw93 朋友卡比做的，内置了小红书、知乎、Twitter/X、Bilibili 等几十个站点的 CLI 适配器。

### 典型场景

1. **小红书调研** - 让 Claude Code 调 opencli xiaohongshu 抓数据，再做分类和热词提炼
2. **舆情汇总** - 把 Twitter/X、Reddit、HackerNews 几个适配器组合起来，同一关键词在多个平台的讨论自动拼成一份日报
3. **没适配的网站** - 用浏览器原语命令描述一遍流程，比如开页面、输关键词、抓表格，Claude Code 自己拼出来

### Routines 功能

能把一段工作流存到云端，按定时、Webhook 或 GitHub 事件自动触发。「周一早上自动跑一遍周报流程」这种事它能接管。

## CLAUDE.md：先把项目背景写清楚

很多人装好之后直接开问，结果每次都要重复交代背景，用一会就觉得很烦。原因几乎都一样：**没建 CLAUDE.md**。

CLAUDE.md 放在项目根目录，Claude Code 每次启动都会先读它，相当于你给新来的同事写的项目交接文档，区别是它每次都会从头认真读一遍而且严格执行。

### 写好 CLAUDE.md 的关键

1. **写得短一点** - 150 行以内为佳，写太长会挤压后续对话的空间
2. **语气直接** - 用命令式，别写软话，"所有注释用中文"比"团队偏好中文注释"有效太多
3. **每条都能判断** - "代码质量要高"没用，"函数超过 50 行必须拆分"才能落地

### 四条最值钱的规则

- **先问清楚再动手** - 别让它猜你的意图，目标说清再写
- **简单优先** - 能两行解决的不写两百行，拒绝过度设计
- **只动该动的** - 不要顺手重构没让它改的代码
- **做完要验证** - 跑通构建和测试才算完，没通过别说完成

### 模板示例

```markdown
# 项目背景
这是一个面向运营同学的客户看板，技术栈 Node.js + Next.js，前端用 React，数据库 PostgreSQL，部署在 Vercel。产品经理是 Alice，设计是 Bob，后端是我自己。

# 工作规范
- 所有注释用中文，变量函数用英文。
- 改动前先说明你打算改什么，确认后再动手。
- 新功能先写实现，不主动加测试，除非我明确要求。
- 数据库表名用下划线分隔，比如 user_profile。

# 禁止项
- 不要主动重构我没提到的文件。
- 不要删除任何文件，除非我明确说删掉。
- 不要在没确认前直接执行 npm install 装新依赖。

# 压缩时保留
长对话被自动压缩时，按优先级保留：
1. 架构决策和它背后的理由
2. 改过哪些文件、改了什么
3. 当前进展状态
4. 还没做完的 TODO
```

### "压缩时保留" 的重要性

Claude Code 的上下文用到一定程度会自动压缩，决策的理由通常是第一个被丢的。比如你之前说过"这里要用 POST 不用 GET，因为数据量大"，压缩之后可能只剩"用 POST"三个字，理由没了。下次再问相关问题，它可能给你一个完全不同的方案，前后矛盾。把这一段写进去，长会话就不会前后打架。

## 需求描述越精确，它越少分叉跑偏

### 模糊版 vs 精确版

**模糊版**：帮我做一个客户跟进工具。

**精确版**：帮我做个销售用的跟进工具，单文件网页存本地。左边列表显示公司名、下单跟进时间、状态，右边详情包括沟通记录、日期、要点。顶部加三个筛选：状态、时间、关键词。数据存浏览器 localStorage，不调后端。

### 验收标准必须给数字

| 无效表述 | 有效表述 |
|---------|---------|
| 页面要快 | 首屏 1.5 秒内 |
| 布局正常 | 在 375 和 1440 两个宽度下不错位 |

### 写需求的原则

- 别用"待定""后续再看""TBD"，Claude Code 碰到这些会自己猜着填，猜的往往不是你要的
- 业务场景先写清楚问题：解决什么、给谁用、怎么算做对了
- 划范围：明确写出"不做登录态、不做分享、不做 SEO，下一期再说"
- 异常情况单独列出来：接口超时怎么办、数据为空展示什么、图片挂了用什么兜底

## 复杂任务先对答案：Plan 与 Auto 模式

复杂一点的任务先用 **Plan 模式**（按两次 `Shift+Tab`），它会先把打算怎么做列出来，方向对了你再让它执行。就像工作场景一样：你不会直接让小李把功能做掉，先拉个会过下方案，觉得 OK 了再动手。

### Auto 模式

按 `Shift+Tab` 循环切到 auto 那一档，Max、Team、Enterprise 都能用，Pro 暂时还没开。它会自己判断：
- 读文件这种安全操作直接跑
- 改数据库、删文件这类风险操作才来问你

刚上手默认开它就行，既不会被无意义的确认打断，也不会让它瞎搞。

## 怎么确认它真的做对了

### 改坏了怎么救回来

- **Git 快照** - 每次大改前让它先跑一遍 `git status` 看清楚都有什么，确认没问题再让它 commit 一个检查点
- **撤销上一步** - 直接对它说"撤销刚才所有改动"，或者按 `/rewind` 回到上一个状态

### 别让它陷入改了试试的死循环

**原因**：没诊断清楚就开始打补丁

**避免方法**：根因没说清楚之前先别动代码。让它先答"问题出在哪个文件的哪一行，为什么会这样"，答含糊继续查，答清楚再改。一上来说"我试试改 X 看行不行"的，直接喊停让它先答根因。

## Max 进阶：alias、模型、长会话

### alias 配置

在 `.zshrc` 里加一行，按 `c` 就直接启动一个不再问权限的 Claude Code：

```bash
alias c='CLAUDE_CODE_AUTO_COMPACT_WINDOW=400000 claude --dangerously-skip-permissions'
```

> ⚠️ `--dangerously-skip-permissions` 不建议刚上手的人用，它字面意思就是"危险跳过所有权限确认"

### 模型策略

用 `/model opusplan` 这个隐藏命令：
- 规划交给 Opus
- 执行交给 Sonnet
- 整体省钱也省时间
- 想更快可以再跑 `/fast`

### 长会话怎么办

- **任务做完就 /clear** - 一个会话只做一件事，做完清掉再开下一件
- **长任务结束前让它写交接笔记** - 直接对它说："把当前进度写成一份 HANDOFF.md"

## Skill 编写指南

Skill 本质就是一个文件夹，放在 `.claude/skills/` 目录下，里面有个 SKILL.md 写清楚什么时候用、要做什么。

### 三种 Skill 类型

**1. 工作流型**：把每次都要做的固定步骤打包

```markdown
---
name: 整理周会
description: 开完会有原始记录需要整理时调用
---
## 输出格式
**本周达成**：[负责人] 完成了什么
**下周计划**：[负责人] 做什么，截止时间
**待讨论**：卡在哪里，需要谁来决定
**行动项**：[谁] [做什么] [什么时候]

## 规则
- 不润色，保持原始措辞
- 信息缺失就标注"待确认"，不要猜
```

**2. 检查清单型**：上线前、发版前、提交前过一遍

```markdown
---
name: 需求上线检查
description: 需求发布前跑一遍，确认没有遗漏
---
## 上线前必须全部通过
- [ ] PRD 里的验收标准逐条确认
- [ ] 设计稿和实现对齐，间距、文案、交互没漏
- [ ] 异常状态（空态、报错、超时）都有处理
- [ ] 数据埋点按规范打好
- [ ] 测试环境验证通过

## 输出
每项 Pass / Fail，有 Fail 必须修完再发布。
```

**3. 领域专家型**：把判断框架沉淀进去

```markdown
---
name: 线上问题排查
description: 收到线上告警或用户反馈异常时调用
---
## 收集信息
- 报错截图或错误日志的完整内容
- 影响范围：哪些用户、哪个页面、什么时间开始
- 最近的变更：代码发布、配置修改、数据变更

## 判断矩阵
| 现象 | 优先检查 |
|------|---------|
| 页面白屏 | JS 报错 → 最近发布记录 |
| 接口超时 | 服务监控 → 数据库慢查询 |
| 数据异常 | 最近数据变更 → 上下游依赖 |

## 输出格式
根因 / 影响范围 / 修复步骤 / 验证方式
```

### 写 Skill 的小坑

1. **description 写触发条件，不写功能介绍** - "开完会有原始记录需要整理时调用"比"把会议录音整理成结构化周报"准确率高
2. **一个 Skill 只做一件事** - 别把审查、发布、调试塞在一起

## 用熟之后的几个小习惯

| 习惯 | 说明 |
|------|------|
| **截图比文字快** | 要描述一个界面问题或者想参考某个设计风格，直接丢图比写一段话准多了 |
| **任务拆小一件件来** | 一句话能讲清楚的任务它几乎不会出错，一大坨需求中间任意一步走偏后面就全偏了 |
| **对话跑偏了就重启** | 在已经跑偏的对话里来回纠正它越纠越乱，清掉上下文重说一遍需求往往更快 |
| **第二天接着干先翻 Recap** | `/clear` 后会自动生成会话摘要 |
| **Memory 跨项目记住偏好** | 直接对它说"记住我喜欢先看方案再执行"，它会写到 `~/.claude/memory/` |
| **双击 ESC 改上一条** | 说错了或者它跑偏了，按两下 ESC 就能回到上一条消息修改 |

## 安全考虑点

| 原则 | 说明 |
|------|------|
| **让它先解释再动手** | 在 CLAUDE.md 里加一条："每次执行 Bash 命令或修改文件前，先用一句话解释要做什么" |
| **看不懂的命令先问** | 它要跑一条你没见过的命令别直接放行，先问它这条命令具体做了什么、有什么风险 |
| **生产环境不要拿来练手** | 本地和测试环境随便折腾，但涉及生产数据库、线上配置的操作一定先在测试环境验证 |
| **密钥别直接粘到对话里** | 要配置 API Key、数据库密码这类东西，让它放到环境变量或者 .env 文件里 |
| **能跑不代表安全** | AI 生成的代码可能有漏洞，涉及登录、支付和个人信息的功能，能用 Clerk 或 Stripe 这种现成服务就别让它从零写 |
