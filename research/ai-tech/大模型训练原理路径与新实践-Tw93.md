# 你不知道的大模型训练：原理、路径与新实践

> 作者：Tw93 (@HiTw93)
> 来源：https://x.com/hitw93/status/2040047268221608281
> 整理时间：2026-04-06

## 核心观点

2026 年来看大模型效果真正拉开差距的地方，慢慢不再是预训练本身了，而在它更后面的那一大段：**后训练、评测、奖励、Agent 训练、蒸馏**，每一个步骤都在影响用户实际感受效果。

---

## 一、大模型训练是一条流水线

### 为什么参数量不是决定性因素？

InstructGPT 的经典案例：一个只有 **1.3B 参数**、做过对齐和偏好优化的模型，在人类偏好评测里能赢过 **175B 的 GPT-3**，参数量差了两个数量级，用户最后却更喜欢那个小很多的版本。

**结论**：训练后半段是真的会改写用户感知。

### 训练链路的六层结构

1. **数据层** - 原始数据处理
2. **算法层** - 模型架构与训练方法
3. **系统层** - 分布式训练基础设施
4. **反馈层** - 评测与奖励设计
5. **Agent 层** - 工具编排与环境交互
6. **部署层** - 服务与上线

---

## 二、预训练只是模型底座

### 预训练的作用

- 把语言分布学进去
- 把大规模文本里的知识和模式压进参数
- 给后面的能力激活留出空间

**关键认知**：下一个 token 预测只描述了训练形式，解释不了为什么规模上来之后，模型会突然多出一些之前没有的能力。

### 规模定律的实际应用

- 参数量、训练 token 数和总计算预算之间有配比问题
- 很多模型不是做小了，而是训练量不足
- Chinchilla 给出的数据最优点：8B 参数约 200B tokens
- 但 Llama3 8B 实际用了 **15T tokens**，超出约 75 倍

### Tokenizer 的影响

| 模型 | 词表大小 | 效果 |
|------|----------|------|
| Llama2 | 32K | - |
| Llama3 | 128K | 序列长度压缩约 15% |

词表设计直接影响：
- 推理成本
- 多语言能力
- 中文、代码、数学公式的 token 效率

## 三、数据配方决定模型能力

### 数据工程 = 能力设计

模型看见什么、看不见什么，代码数学百科各占多大比例，这些选择直接影响模型最后形成的能力分布。

### 数据处理漏斗

```
原始数据 → 文本抽取 → 语言识别 → 质量过滤 → 隐私处理 → 安全过滤 → 去重 → 预训练
```

### 关键问题

1. **去重和污染控制** - document-level 和 line-level dedup
2. **Data Mixing Laws** - 不同类型数据的占比决定能力结构
3. **合成数据** - Self-Instruct、DeepSeek-R1 蒸馏轨迹、Qwen/Kimi 系列的合成监督

### 蒸馏的核心逻辑

模型往往要先在更大规模上形成能力，后面才可能把这些能力压缩到更小的模型上。

**案例**：
- DeepSeek-R1-Distill 系列：RL 后的大模型轨迹让 1.5B 到 70B 的 dense 模型都获得收益
- Llama 3.1 405B 用于提升 8B 和 70B 的后训练质量

---

## 四、系统和架构的约束

### 训练前要考虑的问题

- GPU 数量、显存带宽、并行策略、容错和成本
- 能训多大、支持多长上下文、能不能跑更复杂的后训练

### MoE 架构的权衡

**优点**：相近计算量下扩大总参数，控住每个 token 的激活成本

**代价**：路由复杂、负载均衡难、基础设施重

### 长上下文的约束

128K context 会直接改变：
- attention 成本
- batch size
- 训练 curriculum
- 并行策略

### 训练稳定性问题

- Loss spike（训练损失突增）
- 单块 GPU 静默出错
- NVLink 带宽异常
- 节点间通信抖动

**DeepSeek-V3 案例**：
- 全流程约 2.788M H800 GPU hours
- 预训练完成 14.8T tokens
- 没有出现 irrecoverable loss spike，也没有做任何 rollback

## 五、后训练决定用户感受到的差距

### 指令微调 (Instruction Tuning)

改变回答方式，把怎么接任务、怎么组织输出、怎么像个配合的助手这些要求变成监督信号。

### 三种对齐方法对比

| 方法 | 特点 |
|------|------|
| **RLHF** | 先模仿高质量回答，再用偏好比较做强化 |
| **DPO** | 直接从偏好对比里学，不需要单独训奖励模型 |
| **RFT** | 工程上更容易落地，把任务定义、grader 设计和奖励信号放到产品化流程里 |

### DeepSeek-R1 四阶段后训练流程

**阶段 1：冷启动 SFT**
- 先用少量高质量 CoT 数据热身
- 给 RL 一个更稳定的起点
- 把格式和语言一致性收住

**阶段 2：强化学习（可验证领域）**
- 数学、代码、逻辑等可验证领域
- 使用 GRPO 算法
- 以可程序检验的正确性作为奖励信号

**为什么选 GRPO 而不是 PPO？**
- PPO 需要独立的价值网络，工程负担高
- GRPO 对同一提示词采样多个回答，用组内排名替代绝对价值估计
- 不需要独立的价值网络

**阶段 3：拒绝采样微调**
- 把 RL 产生的成功轨迹过滤后转成新的 SFT 数据
- RL 探索出的好轨迹变成下一轮 SFT 的高质量训练样本

**阶段 4：偏好对齐**
- 融入有益性和安全性偏好反馈
- 把模型调整到符合发布标准的助手形态

## 六、Eval、Grader、Reward 重新定义训练目标

### 核心概念

- **Eval**：决定测什么
- **Grader**：决定一次输出怎么变成分数
- **Reward**：决定模型后面会被往哪里推

### 反馈回路

```
任务定义 → Eval → Grader → 优化 → Rollout → 再评测
```

### Verified Rewards

用程序直接验证正确性，在数学、代码、逻辑这些可验证任务里，不再主要依赖人工偏好。

### Reward Hacking 风险

- **Reward Tampering**：直接篡改奖励计算过程
- **Alignment Faking**：对齐伪装，表面合规但隐藏不对齐意图

### ORM vs PRM

| 类型 | 特点 | 成本 |
|------|------|------|
| **ORM**（结果奖励模型） | 只给最终答案打分 | 低，适合先起步 |
| **PRM**（过程奖励模型） | 给中间步骤打分 | 高，通常是 ORM 的数倍 |

## 七、Agent 训练：优化不只是模型本身

### 推理算力扩展

RL 训练在教模型答题之外，还在教模型**分配推理预算**，知道什么时候多想、什么时候该停。

### Harness 的概念

包在模型外层的控制程序：
- Prompt construction
- Memory update
- Retrieval policy
- Context editing
- Tool orchestration

### 工程案例

**Kimi K2.5 PARL**：
- 只训练 orchestrator，把 credit assignment 收束到编排层
- 奖励信号分三类：任务成功、并行分解、完成约束

**Cursor Composer 2**：
- Self-summarization 和 real-time RL
- 把长时 coding session 与生产流量重新接回训练

**Chroma Context-1**：
- 把 prune_chunks 训成策略本身
- 让 context pruning 直接进入检索过程

### Meta-Harness（2026）

把 harness code 本身作为优化对象：
- 同一个底模，只改 harness，可拉出 **6x** 的性能差距
- 在线文本分类比 ACE 高 7.7 个点，context token 用量压到 1/4

---

## 八、发布后训练链路继续跑

### 持续迭代

- 发布的模型只是快照
- 更强的模型持续给下一代产出训练数据
- 生产流量持续回流到训练

### 发布版本选择

最后上线的版本往往是**产品决策**，不是单一指标上表现最强的那个：
- 真实任务结果
- 拒答风格
- 工具稳定性
- 成本和回归风险

---

## 九、如何判断模型为什么变强了

### 三步分析法

1. **先看变化发生在哪一层**
   - 预训练层 vs 后面的训练流程
   - 模型会不会听指令、会不会用工具，常常不是多训一点语料自己长出来的

2. **再看提升来自哪个环节**
   - 权重和训练配方
   - Reward / Eval / Grader
   - Harness code 和 Deployment loop

3. **最后看上线版本在优化什么**
   - 追求更高上限？
   - 压成本、延迟和回归风险？
   - 给某一类场景做专用化？

### 核心结论

> 今天发布的模型只是一个快照，**链路和 harness program 才是持续在跑的产品**。
